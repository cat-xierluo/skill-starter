# 什么是 GitHub？

GitHub 不是单纯“放代码的网站”。

你可以把它理解成：

**代码仓库 + 协作平台 + 版本入口。**

## 先分清 Git 和 GitHub

很多新手一开始最容易混淆这两个词。

| 名称 | 是什么 | 主要作用 |
|------|--------|----------|
| Git | 本地版本控制工具 | 记录改动、提交历史、分支 |
| GitHub | 云端托管和协作平台 | 存仓库、协作、PR、Issue |

一句话：

- `Git` 负责“怎么记版本”
- `GitHub` 负责“把仓库放到网上并和别人协作”

## GitHub 能做什么

### 1. 托管仓库

你可以把本地 Git 仓库同步到 GitHub。

这样做的好处：

- 云端备份
- 换电脑不丢
- 可以和别人协作

### 2. 管理协作

GitHub 不只是上传代码，还可以：

- 开 Issue 跟踪问题
- 开 Pull Request 合并改动
- 做 Code Review
- 看历史提交

### 3. 展示项目

一个项目的说明、版本、讨论、文档，通常都围绕 GitHub 仓库展开。

### 4. 接入自动化

比如：

- 自动测试
- 自动发布
- 自动检查格式

这些能力很多都能围绕 GitHub 做。

## GitHub 里最重要的几个概念

### Repository

仓库，简称 `repo`。

一个项目通常对应一个仓库。

### Commit

提交。你可以把它理解成“这一次改动的存档点”。

### Branch

分支。你可以在不影响主线的情况下单独开发。

### Pull Request

简称 `PR`。

把“某个分支的改动”申请合并到另一个分支时，就会开一个 PR。

### Issue

用来记录：

- Bug
- 待办
- 讨论项
- 需求

### Fork

把别人的公开仓库复制到你自己的账号下，方便你研究或继续开发。

## 一个最小工作流

这是新手最常见、最实用的一条线：

1. 在 GitHub 创建仓库
2. 在本地 `git clone` 下来
3. 改文件
4. `git add` + `git commit`
5. `git push`
6. 到 GitHub 看改动
7. 如果是团队协作，就开 PR

## 公开仓库和私有仓库

### Public

公开仓库。任何人都能看。

适合：

- 开源项目
- 作品集
- 想展示给别人看的代码

### Private

私有仓库。只有你和你授权的人能看。

适合：

- 商业项目
- 内部工具
- 不想公开的实验项目

## HTTPS 和 SSH 是什么关系

这两个词会在 GitHub 使用里反复出现。

它们不是 GitHub 的功能，而是你连接 GitHub 仓库时的两种认证方式。

### HTTPS

常见 URL 长这样：

```bash
https://github.com/用户名/仓库名.git
```

适合：

- 临时机器
- 偶尔使用
- 通过 Token 认证

### SSH

常见 URL 长这样：

```bash
git@github.com:用户名/仓库名.git
```

适合：

- 自己长期使用的电脑
- 日常开发
- 更稳定的推拉代码体验

如果你是长期开发者，通常更推荐 SSH。

## 新手最常见的误区

### 误区 1：GitHub 就等于 Git

不是。没有 GitHub，你照样可以在本地用 Git。

### 误区 2：代码上传 GitHub 就自动“发布上线”

不是。GitHub 是代码托管平台，不等于你的产品已经部署。

### 误区 3：Push 就等于多人协作完成

也不是。团队协作通常还需要：

- 分支
- PR
- Review
- 合并

### 误区 4：删掉本地文件就等于从 GitHub 永久删干净

不一定。Git 有历史，GitHub 也会保留提交记录。

## 初学者应该先会哪几件事

如果你刚开始学 GitHub，先掌握这 5 件事就够了：

1. 注册账号
2. 创建仓库
3. 克隆仓库
4. 提交并推送
5. 开一个 PR

这比先学一堆高级功能有用得多。
